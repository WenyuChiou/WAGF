## Methods

### Broker Architecture and Governance Pipeline

The Water Agent Governance Framework implements a three-layer architecture (Fig. 1): an LLM agent layer, a governance pipeline, and a simulation engine. At each decision step, the framework assembles the information available to the agent, including personal state variables, social observations from network neighbours, and system-level indicators (drought indices, flood forecasts, institutional announcements), bounded to what the agent's role and location would permit access to. The agent produces a structured response comprising a proposed action, natural-language reasoning, and psychological construct labels where applicable. The governance pipeline then evaluates this proposal through a six-step validation sequence:

1. **Schema validation** (binding): Confirms that the required fields are present and well-formed.
2. **Action legality** (binding): Verifies the proposed action belongs to the agent's type-specific action registry.
3. **Physical feasibility** (binding): Checks mass balance, capacity limits, and domain-specific physical constraints.
4. **Institutional compliance** (binding): Validates against policy, social-coordination, and economic rules — drought-responsive shortage tiers, demand-ceiling stabilisers, programme eligibility, curtailment awareness, and compact allocation. In the code base this step is implemented as three validator categories (institutional, social, economic) that share a common "policy rejection" semantics.
5. **Magnitude plausibility** (advisory): Flags proposed changes that fall outside empirically calibrated bounds (e.g., year-over-year demand shifts within ±15%). The magnitude validator is advisory because a subsequent execution step resamples the numerical magnitude from bounded Gaussians, so an out-of-range proposal is corrected downstream rather than rejected outright.
6. **Theory consistency** (binding, for theory-driven domains): For domains with explicit behavioural theory framing (e.g., Protection Motivation Theory, WSA/ACA dual appraisal), verifies alignment between stated reasoning constructs and the proposed action.

This six-step sequence defines the pipeline architecture (the order in which checks execute). Individual validators within each step are classified by domain function — physical, institutional, social, economic, behavioural, or epistemic — as detailed in Supplementary Tables 1 and 2. Binding validators at steps 1–4 and 6 reject a proposal on failure; the advisory validator at step 5 annotates the audit trail without blocking.

When a binding validator rejects a proposal, the governance pipeline returns a structured explanation identifying the violated rule. The agent receives this feedback and may revise its proposal up to three attempts. If the SAME deterministic rule blocks consecutive attempts (i.e., the blocker set does not change between retries), the revision cycle terminates early and the agent's registry default skill executes (maintain_demand for irrigation, do_nothing for flood), ensuring simulation advancement. This early-exit rule prevents wasted LLM calls when the agent cannot satisfy a fixed constraint. Failed proposals, revision attempts, and governance explanations are recorded in structured decision logs for post-hoc analysis.

The governance architecture is fixed code requiring no modification across domains. All domain-specific knowledge is expressed through declarative configuration files that specify (i) the decision vocabulary available to each agent type, (ii) the state variables, decision attributes, and governance-rule bindings, (iii) natural-language persona prompts establishing cognitive framing, institutional context, and behavioural-theory orientation, and (iv) validator stacks, thresholds, and rule severity levels (advisory versus binding). Separating these configuration artefacts from the pipeline code lets us port WAGF to new water problems by editing the configuration alone.

### LLM Configuration and Inference

All experiments used locally hosted publicly available language models via Ollama, a local inference platform. The primary model was Gemma-3 4B (Gemma Team, 2025; Ollama tag gemma3:4b). Cross-scale comparisons covered eight additional configurations spanning three model families: Gemma-3 12B (gemma3:12b) and 27B (gemma3:27b); Gemma-4 e2b, e4b, and 26b (gemma4:e2b, gemma4:e4b, gemma4:26b); and Ministral 3B, 8B, and 14B (a model series by Mistral AI; Ollama tags ministral:3b/8b/14b). With the primary Gemma-3 4B model this yields nine paired flood comparisons total. Temperature (a parameter controlling output randomness; higher values produce more variable responses) was set to 0.8 with top_p = 0.9 (a complementary randomness parameter) and a maximum input length of 8,192 tokens (approximately 6,000 words). The temperature value follows common practice in generative agent studies (Park et al., 2023). Because temperature directly affects output diversity, the absolute values of behavioural diversity reflect this parameter choice. No temperature sensitivity analysis was conducted; the effect of the validators (with-validators minus without-validators) may vary at different temperatures. Future work should systematically test temperature sensitivity across the range [0.2, 1.0]. A structured response format was enforced through formatted text with rule-based preprocessing that corrects common formatting errors before parsing. Fixed random seeds (reported per domain below) ensured reproducibility across runs. No additional training or calibration of the language models was performed; all behavioural shaping occurred through prompt design and governance constraints. Full agent description templates and governance rule specifications are provided in the Supplementary Information.

**Inference environment.** All nine cross-model pairs were run through Ollama 0.5.x on a single Windows 11 workstation with an NVIDIA RTX 5080 (16 GB VRAM). Models up to 14B parameters fit fully in VRAM; the 26B and 27B configurations used Ollama's automatic GPU+CPU tensor-split fallback. Each model used identical stopping criteria (num_predict = 1536, natural stop on end-of-sequence token) and a retry budget of three parse retries for malformed structured output. Per-run wall times ranged from ~30 min for 3B-class models to ~4 h for the 26B configuration on flood experiments. No run was excluded for runtime errors; one disabled-condition run for Gemma-4 e4b (seed 45) was regenerated to recover from an incomplete write.

### Irrigation Domain Setup

The irrigation application represents 78 agents corresponding to Colorado River Simulation System (CRSS) demand nodes across the Upper and Lower Colorado River Basin. Each agent manages a continuous water-allocation decision over a 42-year horizon (simulating conditions from 1981 to 2022). Agent reasoning follows a dual-appraisal framework, Water Shortage Appraisal (WSA) and Adaptive Capacity Appraisal (ACA), derived from fuzzy-quantitative-linguistic persona heuristics that extend the institutional decision logic of Hung and Yang (2021). Agents are characterized by seniority (priority date), storage access, historical demand flexibility, and basin membership (Upper vs. Lower). Five actions define the decision vocabulary: increase demand (large or small), decrease demand (large or small), and maintain demand. In practice, water users submit annual demand requests driven by crop plans, infrastructure capacity, and contractual obligations rather than explicitly choosing to "increase" or "decrease." The five discrete actions represent the net year-over-year direction of this request: an agent that plants the same crop mix and submits a similar water order is functionally maintaining demand, while one that expands irrigated acreage or switches to more water-intensive crops is functionally increasing it. This abstraction parallels how the Bureau of Reclamation tracks annual changes in consumptive use among CRSS demand nodes. At each annual step, the agent selects an action (the qualitative direction of demand adjustment); the simulation engine then computes a continuous magnitude via bounded Gaussian sampling (mean from persona heuristics, sigma from cluster parameters) and applies it to the agent's current demand request, clamped to [0, water_right]. The resulting demand request represents the agent's intended water use — the quantity it would consume if supply were unconstrained. Actual water delivery (diversion) equals the demand request reduced by the applicable shortage-tier curtailment ratio. This mirrors real-world operations where the Bureau fulfils requests subject to hydrological availability. Agents observe their fulfilled diversion and updated reservoir conditions before making the next year's decision, closing the feedback loop between individual demand choices and collective water-system outcomes. Twelve validators enforce constraints: 5 physical (mass balance, capacity limits, rate-of-change bounds, supply-gap detection), 1 economic (demand-floor stabiliser), 2 institutional (the demand-ceiling rule, linking individual proposals to aggregate basin state, and drought-severity scaling), 2 social (curtailment awareness, compact allocation), 1 temporal (consecutive-increase cap; disabled in production), and 1 behavioural (WSA-ACA construct-action alignment). See Supplementary Table 1 for the complete validator inventory.

To isolate the governance contribution, we conducted a two-condition comparison:
- **Governed**: Full governance pipeline (12 validators, advisory and binding rule levels, retry loop with early exit). Advisory-rule violations issue feedback but allow execution; binding-rule violations block the proposal and trigger retries (max 3).
- **LLM (no validator)**: Identical agent personas, environment, and magnitude sampling, but with governance validation disabled (no validators, no retry loop). The LLM freely selects any action; physical bounds are maintained solely through the code-level clamp in the simulation engine.

Each condition was run with 5 replicate seeds (42, 43, 44, 45, 46) for a total of 10 irrigation runs. Because magnitude is code-generated in both conditions, the comparison cleanly isolates the governance effect on action selection quality.

### Reservoir Simulation

Agent decisions drive Lake Mead through an endogenous mass-balance model. At each annual timestep, the simulation computes:

Storage(t+1) = Storage(t) + Inflows(t) − ΣWithdrawals(t) − Evaporation(t) − Downstream(t)

where Inflows comprise natural flow at Lee Ferry (base 12.0 MAF/yr, scaled by Upper Basin precipitation from historical CRSS traces for 1981–2022, clamped to [6, 17] MAF) plus Lower Basin tributaries (1.0 MAF/yr fixed); Withdrawals are the sum of all 78 agents' fulfilled diversions from the previous year (simultaneous update: all agents observe the same Mead state at the start of each year, make decisions, and withdrawals affect Mead at the start of the following year); Evaporation is elevation-dependent (0.6–0.8 MAF/yr); and Downstream includes Mexico Treaty deliveries (1.5 MAF/yr) and non-agricultural municipal demand (1.2 MAF/yr). Storage-to-elevation conversion uses a USBR-derived lookup table interpolated across 10 control points (895–1,220 ft). The simulation clamps storage changes to ±3.5 MAF/yr to prevent numerical instabilities.

Shortage tiers activate at elevation thresholds following the 2007 Interim Guidelines: Tier 1 at 1,075 ft (5% curtailment), Tier 2 at 1,050 ft (10%), Tier 3 at 1,025 ft (20%). Shortage tiers reduce all agents' fulfilled diversions by the tier percentage — a simplification of the seniority-based curtailment sequence used in actual Colorado River operations, where junior rights are curtailed before senior rights. Upper Basin diversions are additionally constrained by a minimum Powell release of 7.0 MAF/yr and infrastructure capacity of 5.0 MAF/yr, with proportional reduction when aggregate Upper Basin demand exceeds the effective cap.

The demand-ceiling rule blocks demand-increase proposals when total basin demand exceeds 6.0 MAF. This operates at the governance layer (pre-execution), distinct from the code-level clamp that bounds fulfilled diversions post-execution.

Our simulation uses a simplified single-reservoir representation of the Colorado River Simulation System (CRSS). The following simplifications are deliberate. (i) Only Lake Mead is simulated endogenously; Lake Powell and the additional ten reservoirs in the full Bureau of Reclamation CRSS are absorbed into the inflow series or omitted. (ii) Shortage curtailment is applied as a uniform tier-percentage reduction across all 78 demand nodes rather than sequentially by prior-appropriation seniority, which is the rule actually used in CRSS operations. (iii) All agents observe the same Mead state at the start of each year and their withdrawals update the reservoir state at the start of the following year; this simultaneous-update scheme is standard for annual sociohydrology agent-based models (Loucks and van Beek, 2017) but does not capture sub-annual reservoir routing. (iv) Hydropower constraints, environmental flows, and operational override rules are not represented. The consequence is that elevation and shortage statistics in this paper should be read as stylised institutional dynamics, not as predictive water-supply forecasts. Where reported numbers would otherwise invite a forecast reading (for example, Mead reaching 895 ft), the institutional comparison — minimum elevation with validators versus without — remains the defensible claim.

### Flood Domain Setup

The flood-adaptation application models 100 owner-occupant household agents making protective decisions over 10 simulated years. Agents are initialized from census-tract demographic profiles with heterogeneous attributes: flood zone (HIGH, MODERATE, LOW), income level, and prior adaptation state (none, insured, elevated). Flood events occur stochastically with annual probability 0.2; when a flood occurs, agents experience inundation depths drawn from zone-specific log-normal distributions (HIGH zone: μ=1.2 ft, σ=0.8; MODERATE: μ=0.4, σ=0.5; LOW: μ=0.1, σ=0.3), with spatial correlation within zones but independence across zones. Agent reasoning follows Protection Motivation Theory (Rogers, 1983): threat appraisal (perceived probability x severity) combines with coping appraisal (response efficacy x self-efficacy - response cost) to motivate action selection. Four actions are available: purchase flood insurance, elevate home, relocate, or do nothing. The elevation action is permanently blocked once completed.

The experiment uses a three-condition design:
- **Rule-based PMT**: A stochastic rule-based agent using Protection Motivation Theory with softmax action selection (see Rule-Based PMT Agent below). Establishes the traditional ABM baseline.
- **LLM (no validator)**: LLM with HumanCentric memory (Lin and Buelo, in preparation) and governance validation disabled. Agents receive identical prompt templates, PMT-based response format requirements, and environmental context signals as the with-validators condition; the only difference is that proposed decisions bypass validation and execute directly.
- **Governed LLM**: LLM with HumanCentric memory and full governance validation. The HumanCentric memory system features surprise-weighted encoding (events deviating from expectation receive higher salience), arousal-gated consolidation (high-impact events have greater probability of long-term storage), and exponential temporal decay. All flood conditions (main text and cross-model comparison) use HumanCentric memory with the same governance validators. The memory decay function, emotion-class weights, domain adapter for importance assignment, and year-end reflection mechanism are documented in Supplementary Note 10. The implementation additionally includes a cognitive dual-process (System 1 / System 2) module that is not activated in this study; activation is reserved for follow-up work (see Supplementary Note 10.5).

The governance pipeline enforces three tiers of validation (Supplementary Tables 1 and 2). Four **behavioural governance rules** derived from Protection Motivation Theory enforce coherence between cognitive appraisals and protective action: high threat appraisal blocks inaction (binding), low threat appraisal blocks relocation (binding) and elevation (binding), and low coping appraisal discourages costly protective measures (advisory). Five **domain constraints** enforce physical, financial, and institutional feasibility (e.g., preventing re-elevation of already-elevated homes, checking affordability, disallowing adaptation actions after relocation). Four **epistemic guardrails** detect LLM-specific reasoning errors including fabricated references to neighbours' actions, false references to past events that did not occur, and reasoning that contradicts the agent's known state. The Irrational Behaviour Rate (IBR; see Validation Protocol) is evaluated using the same PMT rules applied post-hoc to executed behaviour, measuring violations that either bypassed the validators (with-validators condition) or were never intercepted (without-validators condition). Each LLM condition was run with 5 replicate seeds (5 with-validators + 5 without-validators per model); the rule-based PMT baseline used 3 seeds.

**Strategy diversity metric.** For both domains, behavioural diversity is measured from each agent's annual action selection — the decision submitted in each yearly round — rather than from cumulative protection states. This ensures methodological consistency: irrigation diversity reflects yearly allocation choices among five actions; flood diversity reflects yearly protective-action choices among four actions (purchase insurance, elevate home, relocate, or do nothing). Elevation is a permanent one-time action; insurance must be actively renewed each year, creating a tradeoff in which choosing elevation implicitly forgoes insurance renewal. The annual-decision metric captures these tradeoffs, whereas cumulative protection states (e.g., an agent who has both insurance and elevation) conflate current-year choices with historical decisions. LLM (no validator) agents occasionally selected composite actions (simultaneously purchasing insurance and elevating); these were split into constituent single-action decisions for diversity computation (k = 4). Sensitivity of the diversity metric to alternative composite-action specifications is reported in Supplementary Table 3.

### Rule-Based PMT Agent

To benchmark LLM agents against the traditional ABM approach, we implemented a stochastic rule-based agent using Protection Motivation Theory. This agent computes utility scores for each action based on threat appraisal (TA; derived from flood zone and prior flood experience) and coping appraisal (CA; derived from income, insurance status, and adaptation state), then selects actions probabilistically via a softmax function (temperature τ = 0.5). Base utilities are calibrated to empirical PMT patterns (Bubeck et al., 2012): do nothing U₀ = 0.6, insurance 0.3, elevation 0.1, relocation −1.5, with flood-experience and grant-availability modifiers that shift utilities toward protective actions after flooding (full parameterization in Supplementary Note 9). The rule-based agent uses the same 100 agent profiles, 10-year horizon, stochastic flood events, and evaluation protocol as the LLM-based experiments. Three independent runs with different random seeds were conducted. Unlike LLM agents (which select one action per year), the rule-based agent can simultaneously recommend insurance and elevation when utility scores for both exceed baseline (42.8% of raw decisions). These composite recommendations are split into constituent single-action decisions for behavioural diversity computation. This comparison isolates the representational contribution of LLM-based reasoning: both approaches use PMT as the behavioural theory, but the rule-based agent maps numerical scores to actions through fixed thresholds, while the LLM agent reasons through natural language within governance boundaries.

### Validation Protocol

We adopted a multi-level validation protocol informed by pattern-oriented modelling principles (Grimm et al., 2005).

**Level 1 — Micro-behavioural coherence.** The Irrational Behaviour Rate (IBR) quantifies the percentage of executed decisions that violate the same PMT rules used in the runtime governance pipeline, applied post-hoc to final behaviour. For the flood domain, IBR counts two violation types: relocating or elevating under low threat appraisal (corresponding to the runtime rules `relocation_threat_low` and `elevation_threat_low`). A third rule — do-nothing under extreme threat with adequate coping — is excluded because governance intercepted all violations before execution (Supplementary Note 5). For the with-validators condition, IBR measures violations that escaped the validators; for the without-validators condition, it measures all PMT-inconsistent behaviour. For the irrigation domain, IBR decomposes into three rule sets: (A) physical impossibilities — proposing demand increases at the water-right cap, below minimum utilisation, during supply gaps (fulfilment < 70%), or during Tier 2+ mandatory curtailment; (B) dual-appraisal incoherence — mismatches between the agent's self-assessed WSA/ACA labels and its proposed action, evaluated against a 25-cell coherence matrix mapping each (WSA, ACA) combination to its theoretically rational action set; and (C) temporal violations — four or more consecutive demand increases during drought (drought index >= 0.7). The construct-action coherence rate (1 - IBR_coherence) provides a direct analog to the flood domain's PMT coherence metric. Both metrics are computed per agent per decision step and aggregated across the simulation. In the main text, we report IBR for both domains. For the flood domain, IBR measures PMT-inconsistent decisions (percentage of decisions violating post-hoc PMT rules). For the irrigation domain, IBR measures the fraction of high-scarcity decisions (agent-reported WSA = H or VH) where the agent proposed a demand increase (null expectation under uniform random action = 40%, since 2 of 5 actions are increases).

**Level 2 — Population-level patterns.** We quantify behavioural diversity using the effective heterogeneity entropy (EHE), a normalized Shannon entropy (Shannon, 1948) over the distribution of actions proposed by agents across all decision periods (see Supplementary Note 4 for formal definition). Specifically, behavioural diversity is computed as H / log₂(k), where H is the Shannon entropy of the action frequency distribution and k is the number of actions in the agent's designed decision interface (k = 4 for flood, k = 5 for irrigation). A value of 0 indicates behavioural monoculture (all agents propose the same action); a value of 1 indicates a perfectly uniform distribution across all available actions. EHE measures cognitive diversity, the breadth of strategies agents consider, rather than the diversity of actions ultimately executed; with validators, proposals rejected by the pipeline revert to a domain-appropriate default (maintain demand for irrigation, do nothing for flood), so executed diversity is lower than proposed diversity by design. Cognitive lifespan T_life is defined as the number of simulated years during which behavioural diversity exceeds 0.4, capturing when a population collapses into repetitive behaviour. For irrigation, aggregate demand ratio against CRSS baseline serves as the primary population-level calibration target. IBR for the irrigation domain measures the fraction of high-scarcity decisions (agent-reported WSA = H or VH) where the agent proposed a demand increase; lower values indicate more scarcity-appropriate behaviour (null expectation under uniform random action = 40%, since 2 of 5 actions are increases).

**Level 3 — Cross-scale comparison.** To assess whether the effect of the validators varies with model scale, we compare IBR and EHE between with-validator and without-validator conditions across nine model configurations (Gemma 3 4B/12B/27B, Gemma 4 e2b/e4b/26B, Ministral 3B/8B/14B) using paired t-tests (df = 4 per model from 5 seeds per condition) and 95% confidence intervals (Supplementary Table 6). Seven of nine IBR reductions reached p < 0.05; the two non-significant pairs (Gemma-3 12B, Gemma-4 e2b) are models with near-zero baseline IBR where there was little room for reduction.

### Framework Extension to Sequence-Level Rules

The governance architecture generalises naturally from point-in-time decision rules (R1, R3, R4) to sequence-level rules operating on an agent's decision trajectory. We define three domain-agnostic temporal rule classes as a complementary diagnostic layer: M1 Appraisal–History Coherence (flags decisions where current threat appraisal is low despite a salient event in recent memory, grounded in the availability heuristic and post-flood risk perception decay literature); M2 Behavioural Inertia (flags multi-year action repetition under environmental volatility, grounded in cognitive inertia and adaptive management review cycles); and M3 Evidence-Grounded Irreversibility (flags year-one irreversible commitments made without prior salient evidence, grounded in real options theory and the precautionary principle). Each rule accepts an agent trajectory and a `DomainTemporalAdapter` providing domain-specific event-salience, irreversibility, and volatility predicates; the rule framework (`broker/components/governance/temporal_rules/`) is domain-agnostic and passes unit tests demonstrating zero triggers under a null adapter. In the present study these rules are evaluated post-hoc across all nine model configurations (Supplementary Note 11, Supplementary Table 7) and serve as a diagnostic signal complementing the runtime validators; live enforcement via the reflection engine's year-end hook is reserved for follow-up work.

### Retry Compliance Pattern Analysis

When a runtime validator intercepts an LLM proposal, the resulting retry may reconfigure the decision along two dimensions: the appraisal label and the chosen action. To characterise how LLM agents resolve validator-triggered conflicts, we classify each retried decision into three compliance patterns: (A) *appraisal downgrade* — threat label is lowered from the high set {H, VH} to a lower set while the originally-proposed action is retained (consistent with motivated reappraisal reducing cognitive dissonance; Festinger 1957); (B) *action upgrade* — threat label is maintained while the action is shifted to a more protective alternative (consistent with the canonical Protection Motivation Theory response of matching action intensity to perceived threat); (C) *joint recalibration* — both dimensions change. Pattern distribution is computed across the retried subset of audited decisions and reported per-model and per-condition (Supplementary Table 8).

### Statistical Analysis

Each experimental condition was replicated with five independent random seeds: seeds 42-46 for irrigation and for flood (both with-validators and without-validators). This ensemble size is consistent with standard practice in agent-based water-resource modelling, where individual runs aggregate hundreds to thousands of agent decisions and replicate seeds serve as stability checks on stochastic initialisation rather than formal hypothesis tests (Schlüter et al., 2017). Each seed generates 1,000 agent-year decisions (100 agents × 10 years) for flood and 3,276 (78 agents × 42 years) for irrigation. We report means ± s.d. across seeds and 95% confidence intervals from the paired t-distribution (df = 4); given the small sample sizes, these should be interpreted as exploratory effect estimates rather than confirmatory hypothesis tests. Effects on IBR were directionally consistent across every seed for all nine completed model configurations (5 seeds per condition per model). For agent-level analyses, we computed chi-squared tests on aggregate action distributions pooled across seeds, which provide adequate power given the large number of agent-decisions per condition (3,000 for flood, 16,380 for irrigation). To decompose the contribution of the validators, we computed the construct-action coherence rate on proposed actions (pre-validation) and final actions (post-validation), yielding a per-decision improvement metric. All analyses were conducted in Python 3.11 using NumPy, SciPy, and pandas.
