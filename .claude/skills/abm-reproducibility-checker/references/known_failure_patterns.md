# Known WAGF Reproducibility Failure Patterns

Living catalogue. Each entry documents a real failure that happened in
the project, the detection rule, and the fix. Add new entries here when
new failure modes are caught.

## P1: Hidden CLI flag changes prompt structure

**Date discovered**: 2026-04-19
**Failure**: `--use-priority-schema` flag was silently enabled for the
Gemma-4 V1 batch (3 models × 2 conds × 5 seeds = 30 runs). The flag
injected `### [CRITICAL FACTORS (Focus Here)]` into the user prompt,
which for small LLMs functioned as a directive and shifted Y1 action
distributions by >40 pp.

**How it slipped through**: the flag was default-off; the bat for those
3 models accidentally enabled it; no manifest captured the flag list;
no test compared prompts across runs.

**Detection rule**:
1. For every run, parse the first record of
   `<run_dir>/raw/household_traces.jsonl` and check the `input` field
   for `[CRITICAL FACTORS` substring.
2. Cross-check the bat that produced the run for any flag whose
   default-state is OFF (`--use-priority-schema`, `--persona-style`,
   `--coalition-mode`, etc.).
3. Compare the V1 vs V2 manifest `prompt_flags` field; if either is
   missing, demote to YELLOW pending bat-file inspection.

**Fix applied**: 2026-04-23 archive contaminated dirs to
`examples/single_agent/results/.archive_contaminated_v1_2026-04-23/`,
rerun under V1-matching prompt regime, add startup warning + framework
invariant + CI test against the flag.

**Anti-recurrence**: schema expansion to capture `prompt_flags` in
manifest (item F8 in audit summary).

## P2: Directory name suggests one code state, data was generated by another

**Date discovered**: 2026-04-25
**Failure**: 3 of 5 irrigation Gemma-3 4B baseline seeds (42, 43, 44)
were generated BEFORE commit `4be5092` (v21 `request`-base fix landed
2026-03-03 19:38). Their directory `production_v21_42yr_seed*` correctly
follows v21 naming convention, but the data inside was produced by v20
code (the irrigation_env.py change had not yet landed). Paper EDT1
text claimed "5 seeds (42-46)" — actual data was 3 v20 + 2 v21.

**How it slipped through**: directory naming was applied proactively
("we will generate v21 data into this dir"); the bat ran from a git
checkout that pre-dated the v21 fix; MEMORY.md said "v21 results
COMPLETE: 5 seeds each" reflecting dir count rather than code state.

**Detection rule**:
1. For each run, read manifest `git_commit`.
2. Run `git log --follow <relevant_code> -- <commit>..HEAD` to find any
   commit AFTER the manifest's `git_commit` that touches the relevant
   code.
3. If any such commit is named in commit messages with version tags
   matching the dir name (e.g., dir `production_v21_*` and commit
   message "v21 fix"), the run is misdated. Flag RED.

**Fix applied**: 2026-04-25 archive pre-fix dirs to
`_archive_pre_v21fix_2026-04-24/`, re-run seeds 42-44 with post-fix
code into canonical paths.

**Anti-recurrence**: rename convention should embed
`<git_commit_short>` rather than version tag (e.g.,
`production_4be5092_seed42`).

## P3: Constant audit columns mask missing upstream signal

**Date discovered**: 2026-04-19 (memory pipeline leak audit)
**Failure**: `mem_top_emotion` was constant `"neutral"` across every
production CSV because the memory engine was emitting a sentinel
default rather than the real per-memory emotion. Downstream temporal
diagnostic M1 returned 0 across all models because it depended on real
emotion classification.

**How it slipped through**: sentinel defaults look indistinguishable
from real "neutral" classifications; no CI test asserted column
variance.

**Detection rule**:
1. Run `broker/components/analytics/audit.py:detect_audit_sentinels_in_csv`
   on every audit CSV.
2. Any column constant across ≥80% of rows is flagged.
3. Cross-reference with `broker/INVARIANTS.md` Invariant 4 reserved
   list. Flagged but not on reserved list = RED.

**Fix applied**: memory engine refactor 2026-04-19 (commit `6cd9d2b`)
+ sentinel detector + INVARIANTS.md + CI test.

**Anti-recurrence**: skill always runs sentinel sweep before issuing
GREEN.

## P4: Paper number does not match recomputed value

**Date discovered**: 2026-04-24 (audit Phase 3 spot-recompute)
**Failure**: SI Table 6 Gemma-3 12B row showed `n=4, ΔIBR=+0.25,
p=0.139`. Recomputing at n=5 (all 5 seeds present on disk) gives
`ΔIBR=+0.26, p=0.056`. Different statistical significance level. Same
risk for Gemma-3 27B and Ministral 14B (all manuscript n=3, all 5 seeds
on disk).

**How it slipped through**: SFig 1 forest plot was updated to n=5 in a
later commit; SI Tables were not. No automated check that text values
match plot values.

**Detection rule**:
1. Identify every paper-text numeric claim with model + condition +
   metric + n.
2. For each claim, recompute from the on-disk data using the script
   referenced in Methods.
3. Flag if recomputed value differs by more than the claim's stated
   precision.

**Fix applied**: 2026-04-25 commit `66a3a95` recomputed SI Tables 5/6
at n=5; updated Methods + Results + Discussion "seven-of-nine" →
"eight-of-nine".
